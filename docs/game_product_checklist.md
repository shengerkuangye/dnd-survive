# 游戏成品方向确认清单

## 1. 游戏类型

- 这是一个单机文字/数值驱动的地牢生存经营 RPG。
- 玩法结构参考《地下城堡 2》，尤其是地图探索、有限视野、队伍出征和营地经营。
- 重点不是动作操作，而是队伍培养、资源管理、探索、回合制战斗和长期成长。

## 2. 核心体验

- 玩家从一个营地或据点开始。
- 玩家掌控的是一个营地，而不是固定的一支小队。
- 玩家可以招募大量角色，角色总上限较高，目标上限暂定约 60 人。
- 每次冒险只能派出 4 名角色，避免战斗和操作过于复杂。
- 通过派遣 4 人冒险队外出探索获得资源、装备、经验和剧情线索。
- 探索会消耗资源，比如食物、火把、药品、金币。
- 玩家需要在“继续冒险”和“回营补给”之间做判断。
- 游戏节奏偏慢，强调规划、积累和风险控制。

## 3. 世界观方向

- 整体是 DND 风格奇幻。
- 有地下城、荒野、废墟、古墓、怪物、职业、装备、法术。
- 可以有原创世界观，不直接照搬 DND 官方设定。
- 氛围偏黑暗奇幻、生存、探索，而不是轻松卡通。

## 4. 职业系统

- 职业设计参考 DND。
- 预计会有战士、牧师、游荡者、法师等基础职业。
- 每个职业都有对应的职业建筑。
- 只有营地中建造了对应职业建筑，才可以招募该职业角色。
- 升级职业建筑可以解锁该职业序列的高阶晋升。
- 职业成长不是单纯角色升级，还与营地建筑等级绑定。
- 每个职业有不同定位：
  - 战士负责承伤和物理输出。
  - 牧师负责治疗、祝福、对抗亡灵。
  - 游荡者负责暴击、闪避、探索收益。
  - 法师负责法术伤害和特殊效果。
- 后期通过职业建筑升级扩展职业进阶，但初期先做少量职业和一条简单晋升链。

## 5. 数值系统

- 使用 DND 风格属性：
  - STR 力量
  - DEX 敏捷
  - CON 体质
  - INT 智力
  - WIS 感知
  - CHA 魅力
- 战斗核心使用简化 d20 规则。
- 命中大致是：

```text
d20 + 攻击加值 >= 目标 AC
```

- 伤害可以用骰子表达：

```text
1d6+2
1d8+3
2d4
```

- 不一开始做完整 DND 规则，只取适合游戏的部分。

## 6. 战斗系统

- 战斗使用回合制。
- 每一回合开始时，玩家为队伍成员配置本回合动作。
- 动作可以包括：
  - 普通攻击
  - 使用职业技能
  - 施放法术
  - 防御
  - 使用物品
  - 撤退或尝试撤退
- 玩家可以在本回合内调整队伍成员动作。
- 战斗界面必须有“回合结束”按钮。
- 只有玩家点击“回合结束”后，系统才会结算本回合。
- 如果有队伍成员还没有配置动作，点击“回合结束”时需要提示玩家确认或自动使用默认动作。
- 敌人行动由 AI 或规则自动选择。
- 行动顺序可以使用 DND 风格先攻，简化为：

```text
d20 + DEX 修正
```

- 战斗过程显示日志，比如谁命中、谁闪避、造成多少伤害、谁释放技能。
- 战斗结果影响 HP、资源、经验、掉落。
- 敌人意图不可见，玩家不能提前看到敌人下一回合准备攻击、施法或防御。
- 角色可能受伤、濒死或死亡。
- 角色可以复活，复活功能可以绑定神殿、特殊资源、道具或任务条件。

## 7. 队伍系统

- 玩家管理整个营地中的角色名单。
- 营地角色上限较高，目标上限暂定约 60 人。
- 冒险队伍由 4 名角色组成。
- 每次出征前，玩家从营地角色名单中选择 4 人组成冒险队。
- 每个角色有职业、等级、属性、生命、装备、技能。
- 可以招募新角色。
- 角色可以升级、换装备、学习技能。
- 初期可以先做少量可招募角色和固定 4 人出征规则，后期再做更完整的角色池和替换。

## 8. 营地系统

- 营地是主界面和安全区。
- 玩家在营地里查看资源、角色名单、出征队伍、建筑、任务。
- 后期营地可以有建筑：
  - 酒馆
  - 铁匠铺
  - 神殿
  - 训练场
  - 仓库
  - 炼金工坊
  - 各职业建筑
- 酒馆负责招募角色。
- 职业建筑决定哪些职业可以出现在酒馆招募池中。
- 职业建筑升级后，可以解锁对应职业序列的高阶晋升。
- 仓库负责资源和物品存储。
- 铁匠铺负责装备打造、强化或修理。
- 炼金工坊负责药剂、材料转化或特殊消耗品制作。
- 建筑可以升级，解锁功能或提高效率。
- 初期先做基础营地界面。

## 9. 探索系统

- 冒险形式参考《地下城堡 2》。
- 玩家选择一个地图进行探索。
- 游戏中有多个地图。
- 每个地图有独立的地形、怪物、事件、资源、Boss 和解锁条件。
- 地图探索具有有限视野。
- 玩家只能看到当前视野范围内的地图信息。
- 玩家可以在地图上看到部分怪物的存在。
- 某些敌人或事件可能不可见，导致伏击。
- 区域有推荐等级、资源消耗、怪物池、掉落池、事件池。
- 探索可能触发：
  - 战斗
  - 宝箱
  - 陷阱
  - 资源点
  - 剧情事件
  - Boss
- 区域可以有探索进度。
- 探索进度达到一定程度后解锁新区域或 Boss。
- 地图之间可以通过主线进度、Boss 击败、建筑等级或特殊道具解锁。

## 10. 资源系统

- 游戏会有基础资源。
- 可能包括：
  - 食物
  - 金币
  - 木材
  - 矿石
  - 药草
  - 火把
  - 材料
- 探索消耗资源。
- 战斗和事件产出资源。
- 建筑、装备、治疗、招募消耗资源。
- 资源管理是主要压力来源之一。

## 11. 装备与物品

- 有武器、防具、饰品、消耗品。
- 装备影响 AC、攻击、伤害、属性、技能效果。
- 物品可以来自探索、掉落、打造、商店。
- 初期可以只做武器和护甲。
- 后期再做稀有度、词条、套装、附魔。

## 12. UI 方向

- 使用 Godot 的 Control / Container 系统构建 UI。
- 界面偏功能型、清晰、可编辑。
- 你希望能在 Godot 编辑器里通过节点树和 Inspector 修改 UI。
- 不希望大量 UI 每次启动由脚本动态生成。
- 关键界面应该拆成独立 `.tscn` 场景。
- 整体设计逻辑要符合 Godot 制作游戏的方式。
- 场景结构、UI 布局、容器关系、按钮、列表、面板等优先在 Godot 编辑器里搭建。
- 脚本主要负责状态切换、信号响应、数据绑定和业务逻辑。
- 不要把项目做成“所有 UI 都由脚本运行时 new 出来”的结构。
- 允许动态生成的内容仅限适合重复实例化的数据项，例如角色卡、物品格、地图格、战斗日志行、招募候选项。
- 动态生成时也应优先实例化预制 `.tscn` 组件，而不是纯脚本拼节点。
- 第一阶段先不做完整视觉表现。
- 需要预留视觉接口，方便后续加入角色头像、怪物图标、地图图块、区域插图、技能图标等资源。

## 13. 项目结构方向

- 项目会拆成多个场景：
  - `cover.tscn`：开屏封面
  - `main.tscn`：主入口
  - `game_root.tscn`：游戏主容器
  - `camp_screen.tscn`：营地界面
  - `tavern_screen.tscn`：酒馆招募界面
  - `building_screen.tscn`：建筑管理界面
  - `party_screen.tscn`：队伍界面
  - `roster_screen.tscn`：营地角色名单界面
  - `expedition_screen.tscn`：4 人出征编队界面
  - `map_screen.tscn`：地图探索界面
  - `battle_screen.tscn`：战斗界面
  - `combat_action_panel.tscn`：每回合动作配置界面
  - `end_turn_button.tscn`：回合结束按钮或按钮组件
- 脚本负责逻辑。
- 场景负责 UI 结构。
- 数据尽量放到配置文件里。

## 14. 数据设计方向

- 职业、职业建筑、角色、怪物、物品、地图、地图事件、建筑尽量数据驱动。
- 初期可以用 JSON。
- 后期如果需要更 Godot 化，可以迁移到 Resource / `.tres`。
- 目标是以后改数值时不需要频繁改代码。
- 具体数值暂时先不讨论。
- 当前阶段优先设计工作流程、系统边界、数据结构和开发顺序。

## 15. 成品目标

- 成品不是 DND 规则模拟器。
- 成品是一个使用 DND 风格数值和职业的地下城堡 2 式营地探索 RPG。
- 玩家长期目标是：
  - 建设营地
  - 招募并培养大量角色
  - 组建 4 人冒险队
  - 探索多个有限视野地图
  - 击败 Boss
  - 解锁世界秘密
- 游戏是长期存档养成，不是 Roguelike 式反复开局。
- 游戏需要 3 个存档槽位。
- 3 个存档槽固定命名为：
  - 存档 1
  - 存档 2
  - 存档 3
- 自动存档参考《博德之门》式逻辑：在关键节点自动保存，而不是每一步都保存。
- 自动存档触发点可以包括：
  - 进入新地图
  - 开始重要战斗前
  - 击败 Boss 后
  - 完成关键事件后
  - 返回营地后
  - 建筑升级或重要招募完成后
- 玩家可以手动保存。
- 玩家可以加载之前的存档。
- 保存和加载入口在游戏进行中按 `Esc` 打开的暂停菜单里。
- 暂停菜单需要包含：
  - 保存
  - 加载
  - 保存并退回到主界面
- 存档的具体覆盖、自动存档槽、回滚限制等机制暂时不讨论，当前只确认工作流程。
- 主界面需要有设置入口。
- 设置先包含：
  - 音量
  - 界面大小
- 界面大小先提供几种常见固定尺寸。
- 游戏窗口不能被玩家随意拖拽调整大小。
- 游戏核心乐趣来自：
  - 队伍搭配
  - 资源规划
  - 职业成长
  - 回合行动决策
  - 探索风险
  - 战利品反馈

## 16. 仍需确认的问题

- 职业是严格按 DND 六属性和 d20 来，还是只保留 DND 味道、数值自由调整？
- 复活成本参考《博德之门》式逻辑，具体数值后续再定。
- 手动保存、加载存档、自动存档之间的具体交互机制后续再定。

## 17. 开发工作流程

### 阶段 0：确认设计边界

- 维护本确认清单。
- 先确认玩法结构，不急着讨论具体数值。
- 每当出现新的设计决定，先更新文档，再进入实现。

### 阶段 1：整理项目结构

- 拆分基础场景：
  - `main.tscn`
  - `cover.tscn`
  - `game_root.tscn`
  - `camp_screen.tscn`
  - `map_screen.tscn`
  - `battle_screen.tscn`
- 保持 UI 节点可在 Godot 编辑器中直接编辑。
- 使用 Container 组织界面布局。
- 脚本只负责界面切换、信号响应、数据绑定和业务逻辑，不负责大量临时生成 UI。
- 重复项使用可编辑的 `.tscn` 组件实例化，例如角色卡、物品格、地图格和日志行。

### 阶段 2：建立长期存档框架

- 设计 3 个存档槽。
- 存档槽固定命名为“存档 1”“存档 2”“存档 3”。
- 自动存档参考《博德之门》式关键节点保存逻辑。
- 支持玩家手动保存。
- 支持玩家加载之前的存档。
- 具体存档机制暂时不讨论，先保留必要流程入口和接口。
- 每个存档槽保存：
  - 营地资源
  - 建筑状态
  - 角色名单
  - 当前出征队伍
  - 地图探索进度
  - 已解锁地图
  - 已解锁职业和晋升
- 先做保存和读取流程，不急着填完整数据。
- 自动存档先做触发接口，具体触发点可以逐步接入。

### 阶段 3：建立营地框架

- 做营地主界面。
- 做建筑列表。
- 做酒馆入口。
- 做仓库、铁匠、炼金、职业建筑的占位入口。
- 先让每个建筑能打开对应界面，再逐步补功能。

### 阶段 4：建立角色名单和出征流程

- 做营地角色名单。
- 支持较高角色上限，目标约 60 人。
- 做 4 人出征编队界面。
- 玩家从营地角色中选择 4 人进入冒险队。
- 出征前检查队伍人数、状态、资源消耗。

### 阶段 5：建立招募和职业建筑流程

- 酒馆负责招募。
- 酒馆招募池由已建造的职业建筑决定。
- 没有对应职业建筑，就不能招募该职业。
- 升级职业建筑后，解锁该职业序列的高阶晋升。
- 先实现流程规则，不急着做复杂职业数值。

### 阶段 6：建立地图探索框架

- 做多个地图的选择入口。
- 每张地图保存独立探索状态。
- 地图具有有限视野。
- 玩家只能看到已探索或当前视野范围内的信息。
- 地图上可以显示可见怪物。
- 保留伏击接口，允许不可见敌人或事件触发战斗。
- 先做地图移动和探索状态，再做复杂事件。

### 阶段 7：建立回合制战斗框架

- 战斗开始后进入战斗界面。
- 每回合玩家为 4 名队伍成员配置动作。
- 玩家可以调整动作。
- 玩家点击“回合结束”后才结算本回合。
- 敌人意图不可见。
- 敌人行动在结算阶段由 AI 或规则自动决定。
- 战斗结果写入日志。
- 先实现攻击、防御、使用物品等基础动作，再扩展技能和法术。

### 阶段 8：建立死亡与复活流程

- 角色可以受伤、濒死或死亡。
- 死亡角色不能正常出征。
- 复活功能可以先挂到神殿或占位建筑。
- 复活成本参考《博德之门》式高价值复活服务，后续再定具体消耗。
- 先留接口，确保角色状态系统支持复活。

### 阶段 9：预留视觉接口

- 第一阶段不做完整视觉资产。
- UI 和数据结构中预留：
  - 角色头像字段
  - 怪物图标字段
  - 地图图块字段
  - 技能图标字段
  - 物品图标字段
- 初期可以用占位图、纯文字或颜色块。

### 阶段 10：补内容和数值

- 在基础流程跑通后再开始讨论数值。
- 逐步补：
  - 职业
  - 技能
  - 怪物
  - 地图
  - 建筑升级
  - 装备
  - 掉落
- 每次只补一个小系统，并保证能在游戏里完整走通。
