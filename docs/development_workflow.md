# 开发工作流程

本文档用于指导从 0 开始制作本项目。产品方向以 `docs/game_product_checklist.md` 为准；本文档只写开发顺序和每阶段产出。

## 总原则

- 先做流程，再做数值。
- 先做可玩的最小闭环，再扩展内容。
- 界面和场景优先用 Godot 编辑器搭建。
- 脚本负责状态切换、信号响应、数据绑定和业务逻辑。
- 不把所有 UI 都在运行时动态生成。
- 重复项可以动态实例化，但应优先实例化 `.tscn` 组件，例如角色卡、物品格、地图格、日志行。
- 每完成一个阶段，都要能在 Godot 里运行并手动验证。

## 阶段 1：整理基础项目结构

目标：让项目进入适合 Godot 长期开发的结构。

需要创建：

- `scenes/main.tscn`
- `scenes/ui/cover.tscn`
- `scenes/game/game_root.tscn`
- `scenes/ui/camp_screen.tscn`
- `scenes/ui/map_screen.tscn`
- `scenes/ui/battle_screen.tscn`
- `scripts/main.gd`
- `scripts/game/game_state.gd`
- `scripts/game/save_manager.gd`

工作内容：

- `main.tscn` 只作为入口。
- `cover.tscn` 作为开屏封面。
- `game_root.tscn` 作为进入游戏后的总容器。
- UI 先用 Container 搭静态结构。
- 先不要写复杂逻辑。

完成标准：

- 运行项目后能看到封面。
- 从封面可以进入游戏主容器。
- 所有主要 UI 场景能在 Godot 编辑器中单独打开和修改。

## 阶段 2：做存档入口和存档框架

目标：先搭好长期存档的工作流。

需要创建或完善：

- `scripts/game/save_manager.gd`
- `scripts/game/save_data.gd`
- 存档选择 UI

存档规则：

- 固定 3 个存档槽：
  - 存档 1
  - 存档 2
  - 存档 3
- 支持手动保存。
- 支持加载之前的存档。
- 支持自动存档接口。
- 自动存档参考《博德之门》式关键节点保存。
- 游戏进行中按 `Esc` 打开暂停菜单。
- 暂停菜单提供：
  - 保存
  - 加载
  - 保存并退回到主界面
- 主界面提供设置入口。
- 设置先提供音量和几种固定窗口大小。
- 窗口不能被玩家随意拖拽调整大小。

先保存的数据：

- 当前存档槽编号。
- 营地资源占位。
- 建筑状态占位。
- 角色名单占位。
- 地图进度占位。

暂时不讨论：

- 自动存档覆盖机制。
- 回滚限制。
- 存档冲突处理。
- 具体数值。

完成标准：

- 可以选择 `存档 1/2/3`。
- 可以创建或加载一个存档。
- 可以手动保存一次。
- 退出重进后可以读回基础数据。
- 可以按 `Esc` 打开暂停菜单。
- 可以从暂停菜单进入保存、加载、保存并退回主界面的流程入口。

## 阶段 3：做营地主界面

目标：建立玩家掌控营地的核心入口。

需要创建：

- `scenes/ui/camp_screen.tscn`
- `scripts/ui/camp_screen.gd`

营地界面需要入口：

- 角色名单
- 出征编队
- 酒馆
- 建筑
- 仓库
- 铁匠
- 炼金
- 地图

第一版可以只做按钮和占位页面。

完成标准：

- 进入游戏后默认来到营地。
- 营地能显示基础资源占位。
- 营地按钮能切换到不同占位界面。
- UI 节点在 Godot 中可编辑。

## 阶段 4：做角色名单和角色卡组件

目标：建立营地拥有大量角色的基础。

需要创建：

- `scenes/components/hero_card.tscn`
- `scenes/ui/roster_screen.tscn`
- `scripts/ui/roster_screen.gd`
- `data/heroes/`

规则：

- 玩家管理营地角色名单。
- 角色上限暂定约 60。
- 初期先放少量测试角色。
- 角色卡预留头像接口，但第一阶段可以只显示文字。

角色卡显示：

- 名字
- 职业
- 等级
- HP 状态
- 是否可出征
- 是否死亡

完成标准：

- 角色名单界面能显示多个角色卡。
- 角色卡来自可编辑的 `.tscn` 组件。
- 数据改变后 UI 能刷新。

## 阶段 5：做 4 人出征编队

目标：实现每次冒险只能派 4 人。

需要创建：

- `scenes/ui/expedition_screen.tscn`
- `scripts/ui/expedition_screen.gd`

规则：

- 从营地角色名单中选择出征角色。
- 最多 4 人。
- 死亡或不可出征角色不能加入。
- 出征前检查队伍人数。

完成标准：

- 可以从角色名单选择最多 4 人。
- 选择结果保存到当前出征队伍。
- 超过 4 人时 UI 给出反馈。
- 出征队伍能被地图和战斗系统读取。

## 阶段 6：做建筑系统框架

目标：建立酒馆、职业建筑、仓库、铁匠、炼金等营地建筑的规则入口。

需要创建：

- `scenes/ui/building_screen.tscn`
- `scripts/game/building_manager.gd`
- `data/buildings/`

建筑类型：

- 酒馆
- 仓库
- 铁匠铺
- 炼金工坊
- 神殿
- 职业建筑

规则：

- 建筑有等级。
- 建筑可以升级。
- 职业建筑决定酒馆可招募职业。
- 职业建筑升级解锁对应职业序列的高阶晋升。

完成标准：

- 可以看到建筑列表。
- 可以查看建筑等级。
- 可以升级一个测试建筑。
- 建筑状态可以保存和读取。

## 阶段 7：做酒馆招募框架

目标：让招募与职业建筑产生关联。

需要创建：

- `scenes/ui/tavern_screen.tscn`
- `scripts/ui/tavern_screen.gd`
- `scripts/game/recruitment_manager.gd`

规则：

- 酒馆显示可招募候选人。
- 招募池由已建造职业建筑决定。
- 没有对应职业建筑，就不会出现该职业。
- 招募后角色加入营地角色名单。

完成标准：

- 建造或启用某个职业建筑后，酒馆出现对应职业候选人。
- 可以招募角色。
- 招募结果进入角色名单。
- 招募结果可以保存。

## 阶段 8：做多地图探索框架

目标：建立地下城堡 2 式地图探索基础。

需要创建：

- `scenes/ui/map_screen.tscn`
- `scenes/components/map_tile.tscn`
- `scripts/game/map_manager.gd`
- `data/maps/`

规则：

- 有多个地图。
- 每张地图有独立探索状态。
- 地图有有限视野。
- 玩家只能看到已探索或当前视野范围内的信息。
- 地图上可以看到部分怪物。
- 保留伏击接口。

第一版地图可以是简单网格。

完成标准：

- 可以选择地图。
- 可以在地图网格上移动。
- 移动后更新视野。
- 已探索区域会被保存。
- 可见怪物能显示为占位标记。

## 阶段 9：做战斗入口

目标：地图事件可以进入战斗。

需要创建：

- `scenes/ui/battle_screen.tscn`
- `scripts/game/combat/combat_state.gd`
- `scripts/game/combat/combat_resolver.gd`

规则：

- 战斗由地图怪物或伏击触发。
- 战斗读取当前 4 人出征队伍。
- 敌人数据先用测试数据。
- 战斗结束后返回地图或营地。

完成标准：

- 地图上接触怪物后进入战斗界面。
- 战斗能显示我方 4 人和敌方测试怪。
- 战斗结束后能回到原流程。

## 阶段 10：做回合制动作配置

目标：实现玩家控制回合结束的战斗核心。

需要创建：

- `scenes/ui/combat_action_panel.tscn`
- `scenes/components/end_turn_button.tscn`
- `scripts/ui/combat_action_panel.gd`

规则：

- 每回合玩家为队伍成员配置动作。
- 玩家可以修改动作。
- 界面必须有“回合结束”按钮。
- 只有玩家点击“回合结束”后，系统才结算本回合。
- 如果有人没配置动作，需要提示确认或使用默认动作。
- 敌人意图不可见。

第一版动作：

- 攻击
- 防御
- 使用物品占位

完成标准：

- 玩家能为每个成员选择动作。
- 点击“回合结束”后结算一回合。
- 战斗日志显示本回合结果。
- 敌人行动在结算时自动决定。

## 阶段 11：做死亡和复活接口

目标：角色状态支持死亡和复活。

需要创建或完善：

- 角色状态字段
- 神殿或复活入口占位

规则：

- 角色可以死亡。
- 死亡角色不能正常出征。
- 角色可以复活。
- 复活成本参考《博德之门》式高价值复活服务。
- 具体复活数值暂时不讨论。

完成标准：

- 角色死亡后名单中显示死亡状态。
- 死亡角色不能加入出征队。
- 可以通过占位复活入口把角色恢复为可用状态。
- 死亡和复活状态可以保存。

## 阶段 12：预留视觉资源接口

目标：先不做完整视觉，但不堵死后续美术接入。

需要预留：

- 角色头像字段
- 怪物图标字段
- 地图图块字段
- 技能图标字段
- 物品图标字段
- 建筑图标字段

完成标准：

- 当前 UI 可以用文字或颜色块运行。
- 数据结构中已经有视觉资源字段。
- 后续替换图片资源不需要重写系统。

## 阶段 13：补内容和数值

目标：在流程跑通后，再开始做实际内容。

补内容顺序：

1. 基础职业
2. 基础怪物
3. 基础地图
4. 基础装备
5. 基础建筑升级
6. 基础技能
7. 掉落和奖励
8. 复活成本
9. 职业晋升

原则：

- 每次只补一个小系统。
- 每补一个系统，都要能在游戏里走通。
- 数值调整放在流程稳定之后。

## 第一版 MVP 目标

第一版完成后，玩家应该可以：

1. 打开游戏看到封面。
2. 选择或创建 `存档 1/2/3`。
3. 进入营地。
4. 查看角色名单。
5. 从角色名单中选择最多 4 人出征。
6. 进入一个测试地图。
7. 在有限视野地图上移动。
8. 遇到可见怪物或伏击。
9. 进入回合制战斗。
10. 每回合为 4 名角色配置动作。
11. 点击“回合结束”结算。
12. 战斗结束后返回地图或营地。
13. 手动保存并重新加载。
